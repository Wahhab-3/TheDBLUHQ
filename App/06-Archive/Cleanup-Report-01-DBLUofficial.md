## SAFE TO DELETE — 4.0 GB recoverable

### 1. `.npm-cache-local/` — 536 MB
**Path:** `DBLUofficial/WahhabOfficial/.npm-cache-local/`
**Why delete:** This is npm's local package cache. It's regenerated automatically any time you run `npm install`. It has zero unique data — it's just cached copies of npm packages.
**Risk:** None. Running `npm install` in the Website folder will recreate whatever was needed.

### 2. `Website/node_modules/` — 1.2 GB
**Path:** `DBLUofficial/WahhabOfficial/Website/node_modules/`
**Why delete:** node_modules is NEVER stored — it's regenerated from package.json and package-lock.json. This is the #1 rule of any JS project. You have both lock files in the Website folder, so this can be rebuilt instantly.
**Risk:** None. `npm install` recreates it.

### 3. `Website/dist/` — 205 MB
**Path:** `DBLUofficial/WahhabOfficial/Website/dist/`
**Why delete:** This is the compiled/built output of the Next.js website. It's regenerated by running `npm run build`. The actual source code is in `Website/src/` (204 KB — tiny). The dist folder contains 173 files of compiled JS chunks, static exports, and copied assets. All of it is derived from src/ + public/.
**Risk:** None. `npm run build` regenerates it from source.

### 4. `Final Presentation.mov` — 2.1 GB
**Path:** `DBLUofficial/WahhabOfficial/Wahhab/Final Presentation.mov`
**Why delete:** Single video file taking up 2.1 GB. This is likely a school presentation or promo video. Video files this large should NOT live in a Git-synced Obsidian vault. They'll destroy sync performance and blow up the GitHub repo size.
**Recommendation:** Move to Google Drive, iCloud, or an external drive. Keep a link/reference note in the vault instead.
**Risk:** Low — but confirm you have this backed up somewhere else before deleting from here.

### 5. `Strategy/Marketing/.~lock.*` and `~$*` files — tiny
**Path:** `DBLUofficial/WahhabOfficial/Strategy/Marketing/.~lock.Content_Strategy_Execution_Plan.xlsx#` and `~$Content_Strategy_Execution_Plan.xlsx`
**Why delete:** These are LibreOffice/Excel lock files from when the spreadsheet was open. They're temporary artifacts that should have been cleaned up. They serve no purpose and can cause Git conflicts.
**Risk:** None.

## KEEP BUT MOVE TO CLOUD STORAGE — 1.5 GB

### 6. `Wahhab/Music/` — 1.2 GB
**Path:** `DBLUofficial/WahhabOfficial/Wahhab/Music/`
**Contents:** WAV files, MP3s, cover art for all singles (I Have a Dream, It Just Me, Cookout, Hide and Seek, Le3ba, No Sleep, Hashteg, Kenz, A Mans Tear, etc.)
**Why move:** WAV files are 30-50 MB each. These are your actual music masters — critical creative assets, but they should NOT be in a Git-synced vault. Git is terrible with large binary files. These belong on Google Drive, iCloud, or a dedicated external drive.
**Recommendation:** Create a `Music Masters` folder in Google Drive. Move all WAVs and master files there. Keep only cover art JPGs (small) and a reference note in the vault listing each release with a link to the Drive folder.
**Risk of deleting without backup:** HIGH — these are your original music files. Do NOT delete until backed up.

### 7. `Wahhab/Merch Designs/` — 179 MB
**Path:** `DBLUofficial/WahhabOfficial/Wahhab/Merch Designs /`
**Contents:** 99 PNG files — shirt mockups for Opposing Opposites, Unseen, and other designs. Multiple colorways and mockup variants.
**Why move:** Large PNGs, and there are many variants/iterations. Fine for cloud storage, too heavy for Git sync.
**Recommendation:** Move to Google Drive under a `Brand Assets/Merch` folder. Keep only the final/chosen versions in the vault if needed, or just link to the Drive folder.
**Risk of deleting without backup:** MEDIUM — these are design iterations. Not irreplaceable if you have the source files in a design tool, but annoying to lose.

### 8. `Website/public/` — 204 MB
**Path:** `DBLUofficial/WahhabOfficial/Website/public/`
**Contents:** 89 files — cover art images (JPG, WEBP, PNG), music preview files, and static assets for the website.
**Why keep in project:** These ARE the source assets the website needs. They belong with the website project. But they should NOT be in the Obsidian vault — they should stay in a dedicated website Git repo.
**Recommendation:** When you migrate, this whole Website folder (src/ + public/ + config files) should go into its own GitHub repo, NOT into the Obsidian vault. The vault only gets a reference note.
